## 📝 프로젝트 개요

**린미**는 건강기능식품 전문 이커머스 플랫폼이었습니다.  
처음엔 “이커머스면 비슷하지 않을까?” 싶었지만, 도메인을 파고들수록 일반 상품과는 결이 많이 달랐습니다. 성분 구성, 섭취 방법, 병용 주의사항 등 상품마다 챙겨야 할 정보가 제각각이었고, **약사 자문을 바탕으로 데이터 구조를 잡아가는 과정**이 낯설지만 흥미로웠습니다.

## 😭 문제: 상품 데이터는 RDB 단일 스키마로 버티기 어려웠다

상품 데이터를 RDB 하나로 관리하려다 보니 구조적인 문제가 드러났습니다.  
건강기능식품은 **상품 유형마다 필요한 정보의 형태가 달라서**, 단일 스키마로 억지로 맞추면 `NULL` 컬럼이 계속 늘어나는 구조가 되었습니다.

- 어떤 상품은 **성분 함량 정보**가 10개 이상 필요
- 어떤 상품은 해당 필드 자체가 불필요

상품 종류가 늘어날수록 이 문제는 심해질 게 뻔했고, **초기 설계 단계에서 다른 접근이 필요**하다고 판단했습니다.

## 💡 해결: RDB + MongoDB 분리 설계

공통 필드는 RDB로, 구조가 자주 달라지는 상세 정보는 MongoDB로 분리했습니다.

- **RDB**: 가격, 카테고리 등 모든 상품에 공통으로 적용되는 정보
- **MongoDB**: 성분, 섭취 방법 등 상품마다 구조가 달라지는 상세 정보

DB 설계는 각자 맡은 파트에서 초안을 만들고, 팀 전체가 리뷰하며 구조를 다듬는 방식으로 진행했습니다.  
팀 구성은 **리드 개발자 + 백엔드 3인 + 프론트 1인**이었고, 저는 **상품 관리 파트**를 담당했습니다.

## 💻 개발담당 업무

### 성분 데이터 크롤링

필라이즈 및 건강기능식품 관련 사이트를 크롤링해 성분 데이터를 **수집·정제**했습니다.  
수집한 데이터는 **약사 자문을 바탕으로 검수**한 뒤 데이터 모델에 반영했습니다.

### 엑셀 일괄 등록 기능

관리자 페이지에서 **엑셀 업로드로 상품을 일괄 등록**할 수 있는 기능을 개발했습니다.  
상품 수가 많아질수록 수작업 등록의 한계가 분명해 운영팀 입장에서 실용적인 기능이었습니다.

### 파트너사·관리자용 Web API

파트너사에서 사용하는 웹과 내부 관리자 페이지에서 필요한 API를 주로 담당했습니다.

## 🛠 기술 스택

- **Language/Framework**: Java, Spring Boot
- **ORM**: JPA
- **Database**: MySQL, MongoDB

## 🙌 이 경험에서 얻은 것

#### 1. 도메인을 모르면 좋은 설계가 나오지 않는다

약사 자문을 받으며 데이터 구조를 잡는 과정에서, 개발자가 **도메인을 얼마나 이해하느냐가 설계 품질에 직결**된다는 걸 체감했습니다.  
처음엔 성분이나 섭취 방법 같은 정보를 단순히 “텍스트 데이터”로만 봤는데, 약사의 설명을 들으며 각 필드가 실제로 어떤 맥락에서 쓰이는지 이해하게 되었고, 그제서야 구조가 제대로 잡히기 시작했습니다.  
**기술보다 먼저 “이 데이터가 실제로 어떻게 쓰이는가”를 이해하는 것**이 중요하다는 걸 배운 프로젝트였습니다.

#### 2. 내 설계가 틀릴 수 있다는 걸 팀 피드백으로 알았다

처음 상품 데이터 구조를 잡을 때는 MySQL만으로도 충분히 관리할 수 있다고 생각했습니다.  
하지만 팀 피드백 과정에서 상품마다 구조가 달라질 때 스키마가 어떻게 망가지는지에 대한 이야기가 나오면서, 제 설계의 한계가 명확하게 보였습니다.  
혼자였다면 그냥 밀고 나갔을 방향을 팀이 잡아줬고, 그 이후로는 설계를 **완성본이 아니라 “검토받을 안”으로 들고 가는 습관**이 생겼습니다.

#### 3. 크롤링은 수집보다 정제가 더 어렵다

크롤링 자체보다, 수집한 데이터를 **실제 서비스에 쓸 수 있는 형태로 정제**하는 과정이 훨씬 더 많은 손이 갔습니다.  
건강기능식품 성분 데이터는 사이트마다 표기 방식이 제각각이라, 같은 성분인데도 다르게 표기된 케이스를 모아 통합하는 일 자체가 작은 과제였습니다.  
이 경험을 통해 **데이터 품질이 곧 서비스 품질**로 이어진다는 걸 직접 느꼈습니다.
